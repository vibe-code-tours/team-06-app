# Database Schema Verification & Storage Setup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Verify the existing `supabase/migrations/20250706000000_initial_schema.sql` against `feature-spec.md` and `tech-stack.md`, add the two missing pieces (storage buckets, real QR code generation), and write tests proving RLS policies and the four atomic business functions behave correctly for every role.

**Architecture:** The existing migration already defines 9 tables, 7 enums, 5 helper functions, 4 business functions, 7 triggers, ~35 RLS policies, and 4 realtime publications. This plan does not rewrite it — it adds a second migration for storage buckets, patches the QR-code trigger to be honest about what it does, and adds a Jest test suite (using the harness from `test-infrastructure.md`) covering every RLS policy and business function.

**Tech Stack:** PostgreSQL (Supabase), Supabase CLI, `qrcode` npm package (for real QR generation, invoked from the app layer, not the DB), Jest (from `test-infrastructure.md`).

**Depends on:** `2026-07-08-test-infrastructure.md` must be complete — this plan's tests use `tests/helpers/{resetDatabase,seedTestData,supabaseTestClient}.ts`.

## Global Constraints

- Every table has RLS enabled — no exceptions (per `CLAUDE.md`).
- Role checks go through `user_has_role()` / `user_belongs_to_restaurant()` — don't inline ad-hoc role logic in policies (per `CLAUDE.md`).
- Any schema change requires a new migration file in `supabase/migrations/`; never hand-edit a prior migration that's already been applied (per `CLAUDE.md`).
- After a schema change, regenerate types: `npm run types:generate` (per `CLAUDE.md`).
- Atomic operations stay in Postgres functions, not application code — never reimplement locking in JS (per `CLAUDE.md`).
- Order status transitions: `PENDING → ACCEPTED → PREPARING → READY → COMPLETED`, `CANCELLED` reachable from any pre-COMPLETED state (per `feature-spec.md` §3, already implemented in `update_order_status()`).
- Only one `ACTIVE` session per table; new orders against a table with an active unpaid session must be blocked (per `feature-spec.md` §7, already implemented in `create_order_with_session()`).
- **Local Supabase only.** Every command in this plan (`db:reset`, `db:start`, `types:generate`) targets the local dev stack. Only the designated Supabase-project owner runs migrations against the production/staging project (`supabase db push` against a linked remote project), and only via a merged PR — never from a local machine ad hoc. Nobody else touches the Supabase dashboard directly; schema/RLS changes only land via a migration file in `supabase/migrations/` + PR review, so the dashboard's actual state never drifts from what's in git.

```typescript
// ============================================================================
// QR CODE UTILITIES
// ============================================================================

import QRCode from 'qrcode';

/**
 * Generate a QR code data URL for a table's customer menu link.
 * @param restaurantId - Restaurant UUID
 * @param tableNumber - Table number within the restaurant
 * @param baseUrl - Deployment base URL (e.g. https://app.example.com), no trailing slash
 * @returns data:image/png;base64,... string
 */
export async function generateTableQrDataUrl(
  restaurantId: string,
  tableNumber: number,
  baseUrl: string
): Promise<string> {
  const menuUrl = `${baseUrl}/${restaurantId}/${tableNumber}`;
  return QRCode.toDataURL(menuUrl, { errorCorrectionLevel: 'M', margin: 2 });
}
```
